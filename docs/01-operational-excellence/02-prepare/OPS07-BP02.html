OPS07-BP02: 運用準備の一貫したレビューの確保

運用準備レビュー（ORR）を使用して、ワークロードを運用できることを検証します。

公式ドキュメント

リスクレベル

高リスク

このベストプラクティスを確立しない場合のリスクレベル

  • 運用可能性が不明なままローンチ
  • ガバナンス・セキュリティ要件の未確認
  • ワークロードの定期評価不足
  • 同じ根本原因による障害の繰り返し
ORRなしでワークロードをローンチすると、
運用時に予期しない問題が発生し、
重大な影響を受ける可能性があります！

望ましい結果

  • 組織のORRチェックリストとベストプラクティス
  • ワークロードローンチ前のORR実施
  • ライフサイクル全体での定期的ORR実行
  • 教訓のORRプロセスへの組み込み
  • 必要な手順のローンチ時配置
ORRにより、アーキテクチャ、プロセス、
管理のベストプラクティスを確保！

ORRの構成要素

  • アーキテクチャ推奨事項：設計ベストプラクティス
  • 運用プロセス：日常運用手順
  • イベント管理：インシデント対応
  • リリース品質：デプロイメント基準
  • セキュリティ・ガバナンス：コンプライアンス要件
  • CoE分析：障害分析からの教訓
ORRは単なるベストプラクティスではなく、
過去の障害の再発防止策！

実装ステップ

  • ORRプロセス確立：組織全体での採用と経営陣支援
  • チェックリスト作成：ベストプラクティスと要件の文書化
  • ローンチ前実施：一般提供前の必須ORR実行
  • 定期的見直し：ライフサイクル全体での継続実施
  • 自動化統合：AWS Config、Security Hub活用
  • 継続改善：教訓とベストプラクティスの更新
段階的な実装により、
包括的なORRプロセスを構築！

ORR自動化サービス

  • AWS Config：設定ベストプラクティスの自動検出
  • AWS Security Hub：セキュリティ基準の統合監視
  • AWS Control Tower：ガードレールによる自動制御
  • AWS Systems Manager：コンプライアンス監視
