SEC01-BP01 アカウントを使用してワークロードを分離する

マルチアカウント戦略により、環境（本番、開発、テスト）とワークロード間に共通のガードレールと分離を確立します。アカウントレベルの分離は、セキュリティ、請求、アクセスの強力な分離境界を提供するため強く推奨されます。

公式ドキュメント

リスクレベル

高リスク

このベストプラクティスを確立しない場合、以下のリスクが発生します：

  • ワークロードが誤ってアクセスされた場合の影響範囲拡大
  • 異なる機密レベルのワークロードが同一アカウントに混在
  • 組織単位（OU）構造の不適切な定義
アカウント分離は
セキュリティ境界の
基本中の基本！

期待される成果

クラウド運用、無関係なワークロード、環境を別々のアカウントに分離し、クラウドインフラ全体のセキュリティを向上させるアカウント構造を実現します。

  • クラウド運用の完全分離
  • 無関係なワークロードの独立性確保
  • 環境分離（本番・開発・テスト）
  • クラウドインフラ全体のセキュリティ向上
アカウント分離により
強固な境界を実現！

一般的なアンチパターン

以下のパターンは避けるべき設計です：

  • 異なるデータ機密レベルの複数ワークロードを同一アカウントに配置
  • 組織単位（OU）構造の不適切な定義
  • ビジネスニーズと整合しないアカウント構造

実装ガイダンス

AWSアカウントは、異なる機密レベルで動作するワークロードやリソース間のセキュリティ分離境界を提供します。

主要サービス

  • AWS Organizations - アカウント階層管理とOU構造
  • AWS Control Tower - ランディングゾーン構築とガバナンス
  • Service Control Policies - 組織レベルの予防的制御
  • AWS Config - プロアクティブ・検出的制御
Control Towerで
ワンクリックセットアップ！

実装ステップ

  • OU構造設計 - ビジネスニーズ、データ機密性、ワークロード構造に合わせた設計
  • ランディングゾーン作成 - 一貫したセキュリティ・インフラ基盤の構築
  • ガードレール確立 - 必須制御の自動適用とオプション制御の選択実装
  • リージョン制限 - 新規AWSリージョンへのIAMリソース伝播制御
  • StackSets活用 - 承認済みテンプレートからの複数アカウント・リージョン展開
段階的な実装で
確実に構築！

Control Towerの主要利点

AWS Control TowerはAWS Organizationsの上位レイヤーとして、マルチアカウント環境のベストプラクティス設定を提供します。

  • 統合必須セキュリティ制御 - 組織に参加するアカウントへの自動適用
  • オプション制御 - 特定のOU群に対するオン/オフ切り替え
  • Account Factory - 事前承認済みベースライン・設定オプション付きアカウント自動展開

設計原則

  • 階層継承 - 上位OUから下位メンバーアカウントへの制御継承
  • 権限フィルタリング - OU階層下位レベルでの利用可能権限制限
  • 複雑性軽減 - 各メンバーアカウントの望ましいセキュリティ制御達成に必要なポリシー数・複雑性の削減
階層設計で
管理負荷を大幅削減！
