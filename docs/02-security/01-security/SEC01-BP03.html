SEC01-BP03 管理目標を特定および検証する

2025年3月29日

概要

  • 脅威モデルから特定されたコンプライアンス要件とリスクに基づく
  • ワークロードに適用する必要がある管理目標と管理を導き出し
  • 管理目標と制御を継続的に検証
  • リスク軽減の効果測定に役立つ
脅威モデルからコンプライアンス目標を
明確にして、継続的に検証する
サイクルが大切なんだ！

期待される成果

  • ビジネスのセキュリティコントロール上の目標がコンプライアンス要件に一致するように明確に定義
  • 自動化とポリシーを通じて統制が実装・実施
  • 目標達成のために有効かどうかを継続的に評価
  • 監査担当者に有効性を示す証拠をすぐに報告可能
自動化と継続的な評価で
常に最新の統制状況を把握！
監査時にもすぐ証拠を提出できるよ

一般的なアンチパターン

  • 守るべき規制要件、市場の期待、業界標準についての理解が不十分
  • サイバーセキュリティフレームワークと統制目標がビジネス要件とかみ合っていない
  • 実施されている統制が測定可能な方法で統制目標にしっかりと適合していない
  • 統制の有効性の報告に自動化を使っていない
目標と手段がズレていると
安全対策の効果が薄れてしまう！
適切な把握と測定が重要だよ

一般的なサイバーセキュリティフレームワーク

ビジネスの規制要件、市場の期待、業界標準を考慮し、どのフレームワークがニーズに最も適しているかを判断してください。

AICPA SOC 2

サービス組織のセキュリティ、可用性、処理の完全性に関する信頼性基準

HITRUST

ヘルスケア業界向けの包括的なセキュリティフレームワーク

PCI-DSS

支払いカード業界のデータセキュリティ標準

ISO 27001

国際的に認められた情報セキュリティマネジメントシステム標準

NIST SP 800-53

米国連邦情報システムのセキュリティ管理標準

業界に合ったフレームワークを
選べば、「何を守るべきか」が

AWSサービスの活用

AWS Artifact

目標とするフレームワークに沿ったドキュメントやレポートを入手し、AWSと顧客の責任範囲を理解

AWS Organizations - SCP

サービスコントロールポリシー（SCP）を使用して、非準拠のリソース構成やアクションを予防

AWS Config

非準拠のリソースを監視および報告するためのルールを適用し、必要に応じて強制モードに切り替え

AWS Security Hub

サイバーセキュリティフレームワークに合わせた事前定義されたマネージドルールの導入

AWSのサービスを使えば
統制の実装と監視が
ずっと簡単になるんだよ！

統制の種類

予防的統制
  • 問題が発生する前に阻止するための統制
  • 例: サービスコントロールポリシー (SCP)、IAMポリシー、リソースポリシーなどのガードレール
  • 非準拠のリソース構成やアクションを事前に防止
発見的統制
  • 問題を検知するための統制
  • 例: AWS Configのルール、Security Hubの検出、GuardDutyのファインディングなど
  • 非準拠のリソースを監視および報告
是正的統制
  • 問題を修正するための統制
  • 例: AWS Configの自動修復ルール、EventBridgeとLambdaの組み合わせなど
  • 非準拠のリソースを自動的に修正
予防・発見・是正の
3種類の統制をバランスよく
組み合わせるのがポイント！

Security Hubの活用

自社のサイバーセキュリティのフレームワークに合わせて、事前定義されたマネージドルール一式を導入するには、AWS Security Hubの標準の使用を検討します。

Security Hubの主要標準
  • AWS Foundational Security Best Practices (FSBP): AWSのベストプラクティスに基づく基本的なセキュリティ管理
  • CIS AWS Foundations Benchmark: インターネットセキュリティセンターによるAWSセキュリティベンチマーク
  • PCI DSS: 支払いカード業界のデータセキュリティ標準
  • NIST SP 800-53: 米国連邦情報システムのセキュリティ管理標準
Security Hubの補完

Security Hubに希望する統制検出の機能が組み込まれていない場合は、AWS Config コンフォーマンスパックで補完できます。

  • AWS Config コンフォーマンスパック: 特定のガバナンスやコンプライアンス要件に合わせたAWS Configルールの集まり
  • 業界標準（HIPAA、NISTなど）に基づくテンプレートが用意されている
Security Hubの標準を使えば
一気に数百の統制を
簡単に導入できるよ！

実装手順

1. サイバーセキュリティフレームワークの選択

  • 一般的なサイバーセキュリティフレームワークを評価
  • 選択したフレームワークに合わせて統制目標を定める
  • ビジネスの要件や業界特有の要件に適合させる

2. AWS Artifactでのドキュメント入手

  • フレームワークのガイダンスと責任に関する関連文書を入手
  • 責任共有モデルにおいてどの部分がAWS側の責任で、どの部分が自社の責任かを理解
  • AWS Customer Compliance Guidesを参照

3. 予防的統制の実装

  • SCP、リソースポリシー、ロール信頼ポリシーなどのガードレールを使用
  • 非準拠のリソース構成やアクションを防止
  • 予防的統制の有効性を検証

4. 発見的統制の導入

  • 自社の統制目標に沿ったSecurity Hubの標準を評価・導入
  • AWS Config コンフォーマンスパックで補完
  • 最初は監視モードで実行し、動作を確認できたら強制モードに切り替え

5. 継続的な検証と改善

  • 統制の有効性を定期的に検証
  • 検証結果に基づいて統制を改善
  • 新しい脅威や規制要件に応じて更新
フレームワーク選択→文書入手→
統制実装→検証改善の
サイクルをしっかり回そう！

責任共有モデル

AWS環境での統制目標を検討する際には、責任共有モデルを理解することが重要です。

AWS側の責任
  • クラウドのセキュリティ
  • AWSサービスを実行するインフラストラクチャの保護
  • 物理的セキュリティ、ネットワークインフラ、仮想化レイヤーなど
顧客側の責任
  • クラウドでのセキュリティ
  • AWSにデプロイするもののセキュリティ構成
  • アクセス管理、データ保護、OS/アプリケーションパッチ、ネットワーク構成など

AWS Artifactから入手できるフレームワークドキュメントには、各統制における責任分担が詳細に記載されています。

AWSと顧客の責任を
明確に理解することが
効果的な統制の第一歩！

まとめと効果

  • 明確な管理目標: コンプライアンスとビジネス要件に基づく目標設定
  • 効果的な統制: 予防的・発見的統制の適切な組み合わせ
  • 自動化: AWSサービスによる統制の自動実装と評価
  • 継続的評価: 統制の有効性を常に検証し改善
  • 監査対応: 統制の有効性を示す証拠をすぐに提示可能
  • リスク軽減: 体系的なアプローチによるセキュリティリスクの低減
