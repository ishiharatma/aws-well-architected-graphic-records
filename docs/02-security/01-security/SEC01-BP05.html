責任共有モデルの理解

マネージドサービスを活用するには、責任共有モデルにおける自社の責任範囲が狭くなることを理解することが重要です。

サービスタイプ AWS の責任 お客様の責任 EC2 ハードウェア、ネットワーク、基盤設備 OS、アプリケーション、データ、パッチ適用、セキュリティ設定など RDS ハードウェア、OS、データベースエンジン、バックアップ、パッチ適用 データ、アクセス管理、セキュリティグループなど Lambda ハードウェア、OS、ランタイム環境、スケーリング、パッチ適用 コード、ロジック、IAM権限など DynamoDB インフラ、データベースエンジン、スケーリング、バックアップ、パッチ適用 データ、アクセス管理、暗号化設定など

サービスの種類によって責任範囲が異なり、マネージドサービスを使用することで、お客様側の責任範囲を大幅に削減できます。

マネージドサービスほど

総保有コスト（TCO）の考慮

マネージドサービスと自己管理環境を比較する際は、以下の要素を含めた総保有コスト（TCO）を評価することが重要です：

直接コスト
  • インフラストラクチャのコスト（EC2インスタンス、ストレージなど）
  • ライセンス費用
  • マネージドサービス料金
運用コスト
  • 構築・設定にかかる時間と労力
  • パッチ適用とアップデートの管理
  • バックアップと復旧の作業
  • 監視とトラブルシューティング
セキュリティコスト
  • セキュリティ設定と強化の作業
  • 脆弱性管理とパッチ適用
  • 監査とコンプライアンス対応
  • インシデント対応の体制
目に見えるコストだけでなく
人件費や管理コストも含めて
総合的に判断しよう！

実装手順

1. マネージドサービスで置き換え可能なコンポーネントを評価

  • ワークロードを AWS に移行する場合、リホスト、リファクタリング、リプラットフォーム、再構築、交換が必要かを評価
  • 移行の開始時に追加投資を行うことで、長期的には大幅な節約になる可能性

2. マネージドサービスの導入検討

  • 独自のテクノロジーデプロイをインストールして管理する代わりに、Amazon RDSなどのマネージドサービスを導入
  • LambdaやDynamoDBなどのサーバーレスオプションの検討

3. AWS Artifact の活用

  • AWS Artifactの責任ガイダンスを参考にして、ワークロードに対して導入すべきセキュリティ統制を決定
  • 監査アーティファクトをセキュリティ統制の証拠として監査人や規制当局に提出

4. インベントリの管理と最新情報の入手

  • 使用中のリソースのインベントリを保管
  • 新しいサービスやアプローチに関する最新情報を入手して、スコープを縮小する新たな機会を特定
  • AWS Healthのようなツールで更新を追跡し管理
